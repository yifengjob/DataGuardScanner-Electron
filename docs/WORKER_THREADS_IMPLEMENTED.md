## 🎯 核心改进

### 之前的问题

```
主线程（单线程）
├─ 目录遍历
├─ 文件解析（CPU 密集）← 阻塞事件循环
├─ 敏感检测（CPU 密集）← 阻塞事件循环
└─ UI 更新 ← 被阻塞，无响应
```

**问题**：
- Office/PDF 解析是 CPU 密集型操作
- 即使并发数为 2，也会导致 CPU 100%
- 主线程被阻塞，UI 无响应

---

### 现在的架构

```
主线程（UI 线程）                    Worker 线程池
┌─────────────────┐                ┌──────────┐
│  目录遍历        │                │ Worker 1 │ ← Excel 解析
│  并发控制        │← IPC 通信 →   │ Worker 2 │ ← PDF 解析
│  UI 更新         │                │ Worker 3 │ ← Word 解析
└─────────────────┘                └──────────┘
     ↑                                    ↑
  永不阻塞                          CPU 密集型操作
```

**优势**：
- ✅ **主线程永不阻塞** - 只负责任务调度和 UI 更新
- ✅ **充分利用多核 CPU** - M2 芯片有 8 核，可以并行处理 8 个文件
- ✅ **准确性 100%** - 不跳过任何文件
- ✅ **UI 完全流畅** - 扫描过程中可以随时操作

## 🔧 技术细节

### Worker 池配置

```typescript
const cpuCount = os.cpus().length;  // M2 芯片 = 8
const poolSize = Math.min(config.scanConcurrency || cpuCount, cpuCount);
// 默认使用所有 CPU 核心
```

### 并发控制

```typescript
const maxConcurrency = poolSize * 2;  // 允许队列中的任务数是 Worker 数的 2 倍
// 例如：8 个 Worker，最多 16 个任务在等待
```

### 任务调度

1. **目录遍历**（主线程）→ 发现文件
2. **加入队列** → 如果 activeTasks < maxConcurrency，立即执行；否则加入队列
3. **Worker 处理** → 从队列中取出任务，发送到空闲的 Worker
4. **返回结果** → Worker 完成后，通过 IPC 返回结果到主线程
5. **UI 更新** → 主线程接收结果，更新 UI

## 💡 注意事项

### 1. Worker 线程的限制

- ❌ **无法访问 DOM** - Worker 是纯 Node.js 环境
- ❌ **无法直接使用 Electron API** - 只能通过 IPC 通信
- ✅ **可以访问文件系统** - fs、path 等模块可用
- ✅ **可以使用第三方库** - exceljs、pdf-parse 等可用

### 2. 数据传输

- Worker 和主线程之间通过 `postMessage` 通信
- 数据会被序列化/反序列化
- 对于大文本，有一定开销，但可以接受

### 3. 错误处理

- Worker 中的错误会通过消息传递到主线程
- 如果 Worker 崩溃，会记录错误并继续处理其他文件

### 4. 资源清理

- 扫描结束后，调用 `workerPool.destroy()` 销毁所有 Worker
- 避免内存泄漏

### 问题 2：扫描时 CPU 仍然 100%

**可能原因**：
- Worker 数量太多（超过了 CPU 核心数）
- 文件太大，单个文件解析就很慢

**解决**：
- 在设置中将"并发扫描数"调整为 CPU 核心数的一半
- 或者限制最大文件大小

---

### 问题 3：内存占用过高

**可能原因**：
- Worker 线程过多
- 大文件同时解析

**解决**：
- 减少 Worker 数量
- 降低并发数
